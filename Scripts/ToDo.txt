5. Think about the architecture. Here's an issue Claude pointed out:
"Layering inversion. The architecture intends data → display, mediated by
   the manager, with display "driven by" the manager. But in practice the    
   lowest-level visual object (Tile) is what initiates a move by calling into
   the display layer directly, which bubbles up to the manager, which then   
   reaches back down to mutate state living on the Tile itself. That's three 
   hops for one logical action, and the object that should be most "dumb"    
   (the tile) ends up holding the authoritative position state that the top  
   layer reaches in and overwrites. This will get harder to reason about as  
   more tile interactions are added."
